Strategic Thesis

The guerilla bet.

The web was documents. Then it became platforms. We're building the case that it should be documents again — but better designed, more useful, and owned by the person using them.

Anyone can make an HTML file. AI makes it trivially easy. That's not a threat — it's the thesis. We want everyone making HTML tools. We'll even give them The Forge to do it. But ours will be better — better designed, more thoughtful, more polished, more useful. The same way anyone can take a photo but people still buy from photographers who see differently.

The guerilla strategy:

1

Build the library

Ship tools that solve real problems. Not demos. Not prototypes. Finished, beautiful, useful tools that someone sends to a friend because they actually help.

2

Win by design

One recognizable design system across every tool. When someone opens one, they know it's ours. The brand IS the design language. No logo needed.

3

Give away the forge

The Forge lets anyone build HTML tools with AI. We give it away. More tools in the ecosystem = more validation of the format. We're not threatened by forks — we're accelerated by them.

4

Let revenue emerge

Brand recognition → consulting, licensing, premium tools, Doccoin protocol, enterprise deals. Don't force it. Build the gravity first. The money follows the attention.

The Creation Engine

The Forge.

The Forge is an HTML tool that builds HTML tools. It wraps Claude and Azure OpenAI in a guided workflow — prompting, iteration, quality checks, design system enforcement — so that anyone (team members, community, anyone) can produce tools that meet the Open Intranet standard.

Today it's conceptual — a workflow around AI prompting. The guerilla move is to ship The Forge as one of the first tools in the library. It's both a product and the factory.

⚒

V1 — The Prompt Workflow

A single HTML file with a multi-step form: describe your tool → select category → choose design tokens → send to Azure OpenAI API → receive generated HTML → preview → iterate → export. Ships in Week 1–2.

⚒

V2 — Design System Enforcer

Automatically applies the Open Intranet design tokens (fonts, colors, spacing, components) to generated output. Quality gate before export: accessibility checks, mobile preview, offline test. Ships Month 2.

⚒

V3 — Community Forge

Users can submit their Forge creations to the catalog. Peer review. Quality score. The Forge becomes the on-ramp for community contributions. Ships Month 3–4.

⚒

The Strategic Move

By giving away The Forge, we invite competition. More people building HTML tools = more validation of the format = bigger market for us to be the best in. The Forge is a market-maker, not just a product.

Open Questions

What I need
from you.

These are the decisions and inputs that will sharpen the plan. None of them block Day 0 work — but answering them in Week 1 will shape everything that follows.

  1. Domain: What's the brand name and domain? "Open Intranet"? Something else? This needs to lock before the catalog site ships.
  2. Azure credit timeline: When exactly do the credits expire? This determines sprint intensity and whether we need to front-load AI API spend.
  3. Existing tools inventory: Can you share the actual HTML files you've built so far? I need to assess what's shippable today vs. needs redesign to match the design system.
  4. Team capacity: How many humans are available, how many hours/week, and what's their comfort level with the AI-guided workflow? This sizes the sprint targets.
  5. The Forge — API key model: Does Forge V1 use YOUR Azure OpenAI key (you eat the cost) or BYOK? BYOK = zero cost but higher friction. Your key = faster adoption but burns credits.
  6. Catalog hosting: Azure Static Web Apps is the obvious play with your credits. Any preference for GitHub Pages, Cloudflare Pages, or other? Static hosting is cheap/free everywhere.
  7. Design system authority: Is the design language from the Open Intranet vision doc (Instrument Serif, warm paper tones, green/blue/warm accents) the locked direction? Or is it still evolving?
  8. Doccoin timeline: Is the document connection protocol something we actively design in Month 3–4, or is it a future-state vision for Year 2+?
  9. Revenue philosophy: "No revenue targets" — but are there any expenses beyond Azure credit? Domains, other subscriptions, contractor payments? Or is this truly $0 cash outlay?
  10. Legal / IP: Are the tools MIT-licensed? Creative Commons? Proprietary with free use? This affects community contributions and forking behavior.
